- The main app runs on **Vercel** (serverless) — NOT on a DigitalOcean droplet

- A standalone **InfoHub Collector** process runs on PM2 for data ingestion

## DigitalOcean Managed PostgreSQL

**Service**: DigitalOcean Managed Database — PostgreSQL
**Connection Port**: 25060 (DO managed default)
**Connection Mode**: PgBouncer transaction mode
**SSL**: Required (`?sslmode=require`)

### Connection Pool (per Vercel instance)
```
Max connections:       3
Idle timeout:          5 seconds
Connection timeout:    10 seconds
Max lifetime:          120 seconds (recycle every 2 min)
Prepare statements:    Disabled (required for PgBouncer)
```

### Why Managed (Not Self-Hosted Droplet)
- Automatic backups, failover, and patching
- PgBouncer built-in (critical for serverless — limits connection explosion)
- SSL certificates managed automatically
- Monitoring dashboard in DO console

### Tables (22 total)
See `backend-database-schema.txt` for full schema. Key tables:
- `funding_snapshots` — 10-min interval, top 200 symbols
- `oi_snapshots` — 10-min interval
- `liquidation_snapshots` — continuous ingestion
- `users`, `accounts`, `sessions` — NextAuth
- `api_keys` — Public API v1
- `api_cache` — L2 persistent cache (survives cold starts)

### Data Retention
- Snapshots: 30 days (auto-pruned by cron)
- Alert notifications: 7 days
- API cache: Per-entry TTL (600s–3600s)

## VPS Proxy Server

**IP**: `46.101.247.54`
**Port**: `3100`
**Provider**: Likely DigitalOcean (46.101.x.x is a DO IP range)
**Purpose**: CloudFlare bypass proxy for blocked exchange APIs

### Why It Exists
Vercel datacenter IPs are recognized and blocked by CloudFlare's bot protection on certain exchange APIs. A VPS with a residential-like or different IP range can bypass these blocks.

### Blocked Domains Routed Through Proxy
| Domain | Service | Impact |
|--------|---------|--------|
| `www.bitmex.com` | BitMEX API | ~$158M OI missing without proxy |
| `api.gateio.ws` | Gate.io API | ~$1.5B OI missing without proxy |
| `pro.edgex.exchange` | edgeX DEX API | All edgeX data |
| `query1.finance.yahoo.com` | Yahoo Finance | ETF data, stock heatmap |
| `query2.finance.yahoo.com` | Yahoo Finance (fallback) | ETF data backup |

### How It's Used in Code
```typescript
// src/app/api/_shared/fetch.ts
const PROXY_URL = process.env.PROXY_URL || '';

const PROXIED_DOMAINS = new Set([
  'www.bitmex.com',
  'api.gateio.ws',
  'pro.edgex.exchange',
  'query1.finance.yahoo.com',
  'query2.finance.yahoo.com',
]);

function maybeProxyUrl(url: string): string {
  if (!PROXY_URL) return url;
  const { hostname } = new URL(url);
  if (PROXIED_DOMAINS.has(hostname)) {
    return `${PROXY_URL}?url=${encodeURIComponent(url)}`;
  }
  return url;
}
```

### Files Using Proxy
- `src/app/api/funding/exchanges.ts` (4 usages)
- `src/app/api/openinterest/exchanges.ts` (4 usages)
- `src/app/api/spot-prices/exchanges.ts` (3 usages)
- `src/app/api/tickers/exchanges.ts` (2+ usages)
- `src/app/api/etf/route.ts` (Yahoo Finance)

### Proxy Request Format
```
GET https://46.101.247.54:3100?url=https%3A%2F%2Fwww.bitmex.com%2Fapi%2Fv1%2Finstrument
```
The proxy fetches the target URL with its own IP and returns the response.

## Cloudflare Worker Proxy

**Alternative/secondary proxy** deployed on Cloudflare Workers.

### Configuration
**Location**: `cf-proxy/worker.js` and `proxy/worker.js`
**Name**: `infohub-proxy`
**Region Hint**: Western Europe (`weur`)
**Config File**: `proxy/wrangler.toml`
**Deploy Command**: `wrangler deploy`

### Features
- **SQLite Response Cache**: 60-second TTL using Cloudflare Durable Objects
- **Domain Whitelist**: Only allows BitMEX and Gate.io
- **Authentication**: Optional `PROXY_TOKEN` env var for token-based auth
- **Cache Headers**: `max-age=30, stale-while-revalidate=60`
- **User-Agent**: Custom Mozilla string to appear as browser

### Free Tier Limits
- 100,000 requests/day on Cloudflare Workers free tier
- Suitable for exchange APIs with moderate request volume

### Usage
Set `PROXY_URL` to your Cloudflare Worker URL:
```
PROXY_URL=https://infohub-proxy.your-subdomain.workers.dev
```

## InfoHub Collector

**Type**: Standalone Node.js application (NOT part of Vercel deployment)
**Location**: `infohub-collector/` directory
**Purpose**: Background data fetcher and database ingester

### Process Management (PM2)
```javascript
// ecosystem.config.cjs
module.exports = {
  apps: [{
    name: 'infohub-collector',
    script: 'collector.mjs',
    instances: 1,
    max_memory_restart: '256M',
    autorestart: true
  }]
};
```

### Configuration
| Variable | Default | Description |
|----------|---------|-------------|
| `DATABASE_URL` | required | PostgreSQL connection string |
| `INFOHUB_BASE_URL` | `https://info-hub.io` | Main app URL for API calls |
| `HEALTH_PORT` | `3001` | Health check HTTP server port |

### What It Does
- Fetches exchange data on schedules (separate from Vercel crons)
- Ingests data directly to PostgreSQL
- Provides health check endpoint at `http://localhost:3001/health`
- Dependencies: `dotenv`, `node-cron`, `postgres`, `uuid`

### Deployment
Runs on a server (likely the same VPS at 46.101.247.54 or a separate droplet):
```bash
cd infohub-collector
npm install
pm2 start ecosystem.config.cjs
pm2 save
pm2 startup  # Auto-start on reboot
```

## Upstash Redis

**Service**: Upstash Serverless Redis
**Purpose**: Distributed rate limiting for public API (v1)
**Library**: `@upstash/ratelimit`, `@upstash/redis`

### Configuration
```env
UPSTASH_REDIS_REST_URL=https://xxx.upstash.io
UPSTASH_REDIS_REST_TOKEN=AXxxxx...
```

### Rate Limit Tiers
| Tier | Per-Minute | Daily | Redis Key Prefix |
|------|-----------|-------|-----------------|
| Free | 100 | 5,000 | `ih:rl:free` |
| Pro | 500 | Unlimited | `ih:rl:pro` |

### Algorithm
- Sliding window with analytics
- Fallback: In-memory rate limiter when Redis is unreachable

### Why Upstash (Not Self-Hosted Redis)
- Serverless-compatible (HTTP-based, no persistent TCP connections)
- Works with Vercel Edge runtime
- Pay-per-request pricing
- No infrastructure to manage

## Web Push Notifications

**Standard**: Web Push API with VAPID authentication

### Variables
```env
VAPID_PUBLIC_KEY=<65-byte uncompressed EC public key>
VAPID_PRIVATE_KEY=<32-byte EC private key>
VAPID_SUBJECT=mailto:admin@info-hub.io
NEXT_PUBLIC_VAPID_PUBLIC_KEY=<same as VAPID_PUBLIC_KEY>  # Client-side
```

### Flow
```
1. Client: navigator.serviceWorker.register()
2. Client: PushManager.subscribe({ applicationServerKey: VAPID_PUBLIC_KEY })
3. Client: POST /api/push/subscribe { endpoint, p256dh, auth }
4. Server: UPSERT into push_subscriptions
5. Alert cron: webpush.sendNotification(subscription, payload)
```

### Library
`web-push` npm package for server-side push delivery.

### Minimum Viable Infrastructure
To run InfoHub with core functionality, you need **only 3 services**:
1. **Vercel** (hosting + crons)
2. **DigitalOcean PostgreSQL** (data storage)
3. **At least one data API** (CoinGecko free tier)

Everything else degrades gracefully with fallbacks.
