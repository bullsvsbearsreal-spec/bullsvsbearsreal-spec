## Middleware Pipeline

**File**: `src/middleware.ts`

### Request Flow
```
Incoming Request
    │
    ├── Skip if: /api/chat, /api/admin/**, /api/cron/**, /api/telegram/webhook
    │
    ├── Auth Routes? (/api/auth/signup, check-credentials, forgot-password, etc.)
    │   └── Rate limit: 5 req / 15 min per IP (strict)
    │       └── Headers: Cache-Control: no-store
    │
    ├── API Routes? (/api/**)
    │   └── Rate limit: 120 req / 1 min per IP (moderate)
    │
    ├── API v1 Routes? (/api/v1/**)
    │   └── Bearer token format check: /^Bearer\s+ih_.+$/i
    │       └── Full validation in route handlers (v1-auth.ts)
    │
    └── Public Pages (no auth required)
        / /login /signup /forgot-password
        /funding /open-interest /liquidations /options /basis
        /screener /fear-greed /dominance /top-movers
        /market-heatmap /spreads /orderflow /airdrops etc.
```

### Session Cookie Recognition
```
__Secure-next-auth.session-token    (production HTTPS)
next-auth.session-token             (development HTTP)
__Secure-authjs.session-token       (NextAuth v5 alt)
authjs.session-token                (NextAuth v5 alt)
```

### Rate Limit Headers (set on every API response)
```
X-RateLimit-Limit: 120
X-RateLimit-Remaining: 87
X-RateLimit-Reset: 1711584060
```

### Rate Limit Cleanup
In-memory sliding window with cleanup every 5 minutes (removes expired entries).

## Authentication System

**Framework**: NextAuth.js v5
**Strategy**: JWT sessions (no DB lookups per request)
**Adapter**: Custom PostgresAdapter (postgres.js driver)

### OAuth Providers
| Provider | Env Vars | Scopes |
|----------|----------|--------|
| Google | AUTH_GOOGLE_ID, AUTH_GOOGLE_SECRET | profile, email |
| Discord | AUTH_DISCORD_ID, AUTH_DISCORD_SECRET | identify, email |
| Twitter | AUTH_TWITTER_ID, AUTH_TWITTER_SECRET | users.read, tweet.read |

### Credential Provider Flow
```
1. User submits email + password
2. bcrypt.compare(password, user.password_hash)  // salt rounds: 12
3. Check email_verified: if NULL → throw 'EMAIL_NOT_VERIFIED'
4. Check user_2fa: if totp_enabled or email_2fa_enabled → require twoFactorValidated=true
5. Return { id, name, email, image }
```

### JWT Token Payload
```typescript
{
  id: string;          // User UUID
  name: string;
  email: string;
  image: string | null;  // Avoid data URIs (Vercel 32KB header limit)
  role: 'user' | 'admin' | 'advisor';
  iat: number;
  exp: number;
}
```

### Signup Flow
```
POST /api/auth/signup
  │
  ├── Validate: email format, password (8+ chars, 1 upper, 1 lower, 1 digit)
  ├── Rate limit: 5 / 15 min per IP
  ├── bcrypt hash (salt rounds: 12)
  ├── INSERT INTO users (email_verified = NULL)
  ├── Generate 6-digit code (15 min TTL)
  ├── INSERT INTO email_verification_codes
  ├── Send verification email via Resend
  └── Return { requiresVerification: true }
```

### Login Flow
```
POST /api/auth/check-credentials  (Step 1)
  │
  ├── Validate email + password
  ├── Rate limit: 10 / 15 min per email
  ├── Return { valid, emailVerified, requires2FA, methods }
  │
  ├── If requires2FA:
  │   │
  │   ├── POST /api/auth/2fa/challenge  (trigger email code if needed)
  │   │
  │   └── POST /api/auth/2fa/validate  (Step 2)
  │       ├── TOTP: OTPAuth.TOTP.validate({ token, window: 1 })  // ±30s drift
  │       ├── Email: Atomic claim from twofa_login_codes
  │       └── Backup: Timing-safe compare + array_remove
  │
  └── NextAuth signIn() with credentials
```

### Password Reset Flow
```
POST /api/auth/forgot-password
  │
  ├── Always returns success (email enumeration protection)
  ├── Generate SHA256 hex token (1h TTL)
  ├── Invalidate prior unused tokens for same email
  ├── Send reset email: ${BASE_URL}/reset-password?token=${token}
  │
POST /api/auth/reset-password
  │
  ├── Validate token (hash match + not expired + not used)
  ├── Hash new password (bcrypt 12)
  ├── Update users.password_hash
  └── Mark token used
```

### Admin Authorization
```typescript
// Re-verifies DB on every call (prevents stale JWT exploits)
async function isAdmin(userId: string): Promise<boolean>
async function requireAdmin(): Promise<Response | null>          // admin only
async function requireAdminOrAdvisor(): Promise<Response | null> // admin OR advisor
```

## Two-Factor Authentication

### TOTP Setup
```
POST /api/auth/2fa/setup  (session required)
  │
  ├── Block if TOTP already enabled
  ├── Generate: OTPAuth.TOTP(20 bytes, 6 digits, 30s period, SHA1)
  ├── Generate: 8 backup codes (4 random bytes each, uppercase hex)
  ├── Store in user_2fa (totp_enabled = false)
  ├── Return: { qrCode (data URL), secret (base32), backupCodes }
  │
POST /api/auth/2fa/verify  (confirm setup)
  │
  ├── Validate submitted code against stored secret
  ├── Set totp_enabled = true
  └── Store backup codes
```

### TOTP Validation Methods

| Method | Validation | Anti-Replay |
|--------|-----------|-------------|
| TOTP | `OTPAuth.TOTP.validate({ token, window: 1 })` — allows ±30s clock drift | Time-based (window) |
| Email | Atomic DB claim: `UPDATE ... SET used=true WHERE used=false RETURNING *` | Single-use flag |
| Backup | Timing-safe comparison, then `array_remove(backup_codes, code)` | Array removal |

### Rate Limits (per email, in-memory)
- 2FA challenge: 10 / 15 min
- 2FA validate: 5 / 15 min

## Rate Limiting

### Middleware Rate Limits (In-Memory)

| Target | Limit | Window | Key |
|--------|-------|--------|-----|
| Auth routes | 5 req | 15 min | IP address |
| API routes | 120 req | 1 min | IP address |
| Signup | 5 req | 15 min | IP address |
| Forgot password | 5 req | 15 min | IP address |
| Check credentials | 10 req | 15 min | Email (10K key cap) |
| 2FA challenge | 10 req | 15 min | Email |
| 2FA validate | 5 req | 15 min | Email |
| Chat | Custom | Custom | Session/IP |

Implementation: Sliding window with Map<string, {count, firstRequest}>. Cleanup every 5 minutes.

### v1 API Rate Limits (Upstash Redis)

| Tier | Per-Minute | Daily | Redis Prefix |
|------|-----------|-------|-------------|
| Free | 100 | 5,000 | `ih:rl:free` |
| Pro | 500 | Unlimited | `ih:rl:pro` |

Fallback: In-memory when Redis is down.

### v1 API Key Cache
- TTL: 1 minute
- Max size: 1,000 keys (LRU eviction)
- Lookup: Memory cache → DB hash comparison → fail

| liquidation-ingest | Every 1 min | ~15s | liquidation_snapshots |
| alerts | Every 5 min | ~10s | alert_notifications |
| arbitrage-alerts | Every 5 min | ~5s | (Telegram send) |
| news-alerts | Every 15 min | ~5s | (Telegram send) |
| calendar-alerts | Every 1 hour | ~3s | (Telegram send) |
| portfolio-snapshot | 12:00 UTC daily | ~30s | portfolio_snapshots |
| telegram-reports | 08:00 UTC daily | ~10s | (Telegram send) |
| telegram-weekly | 10:00 UTC Sunday | ~10s | (Telegram send) |

### Snapshot Cron (Most Important)
```
GET /api/cron/snapshot
  │
  ├── Check: minute % 10 === 0 (only full runs every 10 min)
  ├── Fetch /api/funding (25s timeout)
  ├── Fetch /api/openinterest (25s timeout)
  ├── Pick top 200 symbols by exchange count
  ├── INSERT INTO funding_snapshots (batch)
  ├── INSERT INTO oi_snapshots (batch)
  ├── Compute spreads from ticker data
  ├── INSERT INTO spread_snapshots
  ├── Track arb_opportunities (open/close)
  └── Prune data older than 30 days
```

### Liquidation Ingestion Cron
```
GET /api/cron/liquidation-ingest
  │
  ├── Binance: GET /fapi/v1/allForceOrders?limit=200
  ├── OKX: GET /api/v5/public/liquidation-orders (per-ULY, 2 req/2s rate limit)
  ├── HTX: GET /linear-swap-api/v3/swap_liquidation_orders (batch 4 contracts)
  ├── gTrade: GET /api/trading-history/24h (filter action='TradeClosedLIQ')
  ├── Deribit: GET /api/v2/public/get_last_trades_by_currency (filter liquidation=true)
  │
  └── INSERT INTO liquidation_snapshots
      ON CONFLICT (symbol, exchange, side, price, ts) DO NOTHING  // dedup
```

### Alerts Cron
```
GET /api/cron/alerts
  │
  ├── Fetch market data (funding, OI, prices)
  ├── For each user with active alerts:
  │   ├── Check cooldown (default 60 min)
  │   ├── Evaluate alert conditions
  │   ├── If triggered:
  │   │   ├── Send email (Resend)
  │   │   ├── Send Telegram (if connected)
  │   │   ├── Send push notification (Web Push)
  │   │   └── Log to alert_notifications
  │   └── Update cooldown
  └── Prune notifications older than 7 days
```

## Error Handling & Circuit Breaker

### Circuit Breaker (Per-Exchange)

```
Configuration:
  FAILURE_THRESHOLD:  5 failures
  WINDOW:             10 minutes
  INITIAL_COOLDOWN:   2 minutes
  BACKOFF:            Exponential (2 → 4 → 8 → 16 → 30 min)
  MAX_COOLDOWN:       30 minutes
  LOG_THROTTLE:       Once per 10 min

State Machine:
  CLOSED (normal)
    ├── Failure count < threshold → stay CLOSED
    └── Failure count >= threshold → transition to OPEN

  OPEN (skip exchange)
    ├── Cooldown not expired → skip (return empty)
    └── Cooldown expired → transition to HALF-OPEN

  HALF-OPEN (retry)
    ├── Success → reset to CLOSED
    └── Failure → back to OPEN (double cooldown)
```

### Exchange Fetch Retry
```
Per-exchange:
  - Attempt 1: Normal fetch
  - On failure: Wait 500ms
  - Attempt 2: Retry
  - On second failure: Log error, return empty, increment circuit breaker
```

### Stale Data Fallback
```
Normal flow:
  1. Try fresh fetch from exchanges
  2. If all fail: serve last cached data (L1 memory)
  3. If L1 empty: serve from L2 (database cache)
  4. If L2 empty: return error response

Response headers indicate staleness:
  X-Cache: STALE         (served from expired L1 cache)
  X-Cache: STALE-RAW     (served from raw cache, not filtered)
```

### Logging
```typescript
logExchangeError(exchange, error, context)   // Warning level + Sentry
logApiError(route, error, context)           // Error level + Sentry
logValidationError(route, issues, context)   // Warning level + Sentry
```

## Caching Architecture

### Three-Layer Cache Strategy

```
Layer 1: In-Memory (per serverless instance)
  ├── Fastest, but lost on cold start
  ├── TTL varies by data type (2s - 30min)
  └── Map<string, { data, expiresAt }>

Layer 2: Database (api_cache table)
  ├── Survives cold starts
  ├── Shared across all instances
  ├── TTL varies (600s - 3600s)
  └── PostgreSQL JSONB

Layer 3: CDN Edge (Vercel)
  ├── Cache-Control headers
  ├── s-maxage (serve cached) + stale-while-revalidate (background refresh)
  └── Closest to user
```

### Request Deduplication
